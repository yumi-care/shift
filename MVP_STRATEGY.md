## 1. Claude Code 「毎回忘れる」問題への対応

### 現状分析

- メモリファイルは保存されている ✅
  - `/memory/project_shift_overview.md`
  - `/memory/MEMORY.md`
- しかし、セッション起動時にこれらが自動ロードされるか **不確定**

### 対策

#### A. ファイルベースの記憶（必ず機能）

```
shift/
├── SPECIFICATIONS.md          ← 最初に読む
├── DEVELOPMENT_RULES.md       ← 次に読む
├── MVP_STRATEGY.md            ← このファイル
├── .claude/CLAUDE.md          ← プロジェクト管理
└── .claude/SESSION_INIT.md    ← 【新規】セッション起動時に読むべき最小限情報
```

**SESSION_INIT.md** に以下を記載：
- プロジェクトの概要（1ページ）
- フェーズ構成（Phase 1 〜 6）
- IMMUTABLE な部分（削除禁止）
- 現在の開発状況
- 次のステップ

#### B. メモリ活用

メモリはセッション開始時に自動ロードされる可能性が高い。
確認方法：セッション起動時に「memory の内容を確認できているか」をユーザーに確認してもらう。

#### C. ユーザーへの指示

セッション起動時：
```
【初動チェックリスト】
□ SPECIFICATIONS.md を読んだか（仕様書）
□ DEVELOPMENT_RULES.md を読んだか（開発ルール）
□ .claude/CLAUDE.md を読んだか（プロジェクト管理）
□ Memory が自動ロードされているか確認

これが完了して初めてコード実装に入る
```

## 2. MVP（Minimum Viable Product）仕様

### 設計原則

> **「動作 > セキュリティ」**

初期段階ではセキュリティを最小限にし、**動作確認と UX** に注力。
セキュリティは Phase ごとに段階的に追加。

### セキュリティレベルの段階化

| Phase | 認証 | マルチテナント | 権限管理 | 入力バリデーション |
|-------|------|---|---|---|
| **1-2** | 簡易ログイン（ハードコード可） | 基本的な分離 | なし | 最小限（型チェックのみ） |
| **3-4** | JWT 導入 | 完全分離 | 基本的な admin/user | 中程度 |
| **5-6** | OAuth2 検討 | 完全分離 | 詳細な権限（拠点別） | 厳密 |

### Phase 1（事業所登録）の MVP

#### セキュリティ・認証
```
❌ OAuth2
❌ メール認証
✅ 簡易ログイン（ユーザー名 + パスワード、DB にハードコード可）
   例）admin / admin123 でテスト
```

#### マルチテナント分離
```
✅ テーブルに corp_id カラム
✅ WHERE corp_id = xxx フィルタ（基本的な分離）
❌ 完全なテナント隔離（後から）
```

#### 入力フォーム
```
✅ テキスト入力（法人名、事業所名）
✅ ドロップダウン（サービス種別）
✅ 数値入力（定員数、営業時間）
❌ バリデーション（初期段階では最小限）
   - 空白チェックのみ
   - 型チェックのみ
   - 桁数制限なし（後から）
```

#### 確認画面
```
✅ 入力内容の確認画面を表示
✅ 「修正」「確定」ボタン
❌ 履歴管理（後から）
❌ 監査ログ（後から）
```

## 3. Phase 1 画面フロー（MVP）

### フロー

```
① ログイン画面
   ↓
② メニュー画面（法人選択 or 作成）
   ↓
③ 事業所登録フォーム（テキスト入力）
   ├─ 法人名
   ├─ 事業所名
   ├─ サービス種別（ドロップダウン）
   ├─ 定員数
   └─ 営業時間
   ↓
④ 入力内容確認画面
   ├─ 「修正」ボタン → ③に戻る
   └─ 「確定」ボタン → DB に保存
   ↓
⑤ 完了画面
```

### 各画面の仕様

#### ① ログイン画面（MVP）
```html
ユーザー名: [admin_______]
パスワード: [●●●●●●●●●]
           [ログイン]

※ 初期段階ではハードコード可
  admin / admin123
```

#### ③ 事業所登録フォーム（MVP）
```html
【事業所登録】

法人名：*
[テキスト入力___________]

事業所名：*
[テキスト入力___________]

サービス種別：*
[▼ 選択してください____]
  ├─ 特別養護老人ホーム
  ├─ グループホーム
  └─ ...

定員数：*
[数値入力: ____ 名]

営業時間：
[開始時刻] 08:00
[終了時刻] 18:00

[確認]  [キャンセル]
```

**バリデーション（MVP）**
```
✅ 空白チェック
✅ 数値チェック（定員数）
❌ 桁数制限
❌ 特殊文字チェック
❌ 住所の詳細バリデーション
```

#### ④ 入力内容確認画面（重要）

```html
【入力内容確認】

以下の内容で登録します。
修正したい場合は「修正」をクリックしてください。

━━━━━━━━━━━━━━━━━━━━
法人名：          ABC福祉法人
事業所名：        ゆうみのいえ
サービス種別：    共同生活援助
定員数：          6 名
営業時間：        08:00 〜 18:00
━━━━━━━━━━━━━━━━━━━━

[修正]  [確定]
```

**重要：** 確認画面で修正を促す → ユーザーエラーを減らす

## 4. Phase 3（シフト提出）の MVP フロー

### フロー

```
① スタッフログイン
   ↓
② シフト提出画面
   ├─ 提出対象月を選択
   ├─ 拠点を選択（複数拠点の場合）
   └─ カレンダーで希望を入力
   ├─ シフト種別を選択 or 希望休
   ↓
③ 入力内容確認画面
   ├─ 「修正」ボタン
   └─ 「提出」ボタン
   ↓
④ 提出完了画面
   └─ 「後で修正する」リンク
   ↓
⑤ 提出後も「マイシフト」から修正可能
   └─ 「このシフトを修正」→ ③ に戻る
```

### 重要な UX

```
✅ カレンダー形式で視覚的に入力
✅ 確認画面で修正を促す
✅ 提出後も修正可能（「マイシフト」画面）
✅ 提出期限をユーザーに表示
❌ 自動保存（初期段階では不要）
❌ 複雑なバリデーション
```

## 6. 初期段階で削除・上書きしてはいけない部分

MVP でもこれだけは守る：

```
⚠️ マルチテナント分離（corp_id フィルタ）
   → 複数企業のデータが混在する危険

⚠️ テーブル基本構造（corp_id, id など）
   → 後から追加困難

⚠️ API エンドポイントの基本設計
   → Phase が進むたびに使用される

⚠️ テスト（削除・スキップ厳禁）
   → 後の修正を難しくする
```

```markdown
⬇︎

これが完了して初めてコード実装を開始
```

## 9. 修正可能性の設計

### 提出後も修正できるようにするための設計

```sql
-- shift_submissions テーブル
CREATE TABLE shift_submissions (
    id INT PRIMARY KEY,
    staff_id INT,
    corp_id INT,
    target_month DATE,
    submitted_at TIMESTAMP,
    last_modified_at TIMESTAMP,  -- ← 重要：修正時刻
    status ENUM('draft', 'submitted', 'modified'),  -- ← 重要：ステータス
    shift_data JSON  -- ← 柔軟な構造
);
```

**フロント仕様：**
```
提出済みシフト表示画面
├─ 「このシフトを修正」ボタン
│  └─ shift_data を再編集可能にする
├─ 修正日時を表示
└─ 「修正を確定」ボタン → last_modified_at を更新
```

## 10. まとめ

### MVP 開発時のプリンシパル

1. **動作優先** → セキュリティは後から
2. **UX 優先** → ユーザーがストレスなく使える
3. **確認画面必須** → 提出前に内容を確認・修正させる
4. **修正可能設計** → 提出後も修正できる
5. **段階的強化** → Phase ごとにセキュリティを追加

### セッション起動時の対応

```
新しいセッション起動
   ↓
Memory + SESSION_INIT.md を読込
   ↓
前回の進行状況を確認
   ↓
実装開始
```

---

**次の実装ステップ：**
1. SESSION_INIT.md を作成
2. 技術スタック確定（TECH_STACK.md）
3. Phase 1 の UI 画面設計詳細化
4. テスト駆動開発で実装開始
