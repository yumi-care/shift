# shift（シフト作成・勤務体制表自動化ツール）

## プロジェクト概要

**プロダクト名：** シフト作成・勤務体制表 自動化ツール  
**形態：** マルチテナント型 SaaS（自社利用 + 外部販売）  
**対象業種：** 介護・障害福祉サービス  
**最終成果物：** Webアプリ（ブラウザで操作・印刷対応）

---

## 開発方針

### フェーズ分割
6つのフェーズに分割し、段階的に実装。各フェーズ完成後に実際に操作して「打感」で修正。

**Phase 1：事業所登録** → Phase 2：スタッフ登録 → Phase 3：シフト提出 → Phase 4：シフト表作成・確認 → Phase 5：実勤怠連携 → Phase 6：勤務体制表生成

### 設計思想
> **「最初の設定だけで、あとは完全自動」**
> - シフト作成・勤務体制表作成に管理者の工数をかけない
> - 勤怠システム（タッチオンタイム）と連動し、実勤怠データをもとに自動生成
> - 管理者がやることは「初期設定」と「例外対応」のみ

## ★ システム設計プロセス（プロの手法）

### 設計フェーズ（実装前に必ず実施）

1. **全体アーキテクチャの決定**
   - フロントエンド・バックエンド・データベースの構成
   - データベーススキーマの確定
   - API仕様の定義
   - 認証・認可の仕組み
   - マルチテナント分離戦略

2. **データフロー設計**
   - Phase 1〜6 各フェーズ間でのデータの流れ
   - どのフェーズでどのデータが生成・参照・更新されるか
   - データの永続化方法（API vs localStorage）
   - 各フェーズのデータソースを統一

3. **UI/UX導線設計**
   - 各ユーザー（管理者・スタッフ）の操作フロー
   - 各フェーズの画面遷移
   - エラーハンドリング
   - 入力データの検証ルール

### テスト段階（開発環境）

**目的：** 機能仕様の検証、UI/UX確認

**構成：**
- フロントエンド：Vite（localhost:5173）
- バックエンド：簡易 Mock API（localhost:8000）
- データストレージ：
  - **管理画面（Phase 1）**：Mock API（インメモリ）
  - **スタッフ登録（Phase 2）**：localStorage（テスト用）
  - **シフト申告（Phase 3）**：localStorage（テスト用）
  - **ダッシュボード**：API + localStorage の統一化

**ルール：**
- テスト段階では localStorage と API を混在させない
- 本番設計に基づいた構造でテストする
- データベーススキーマは本番と同じ仕様

### 本番段階

**構成：**
- フロントエンド：React（本番環境）
- バックエンド：FastAPI or Node.js（本番環境）
- データベース：PostgreSQL（マルチテナント対応）
- 認証：JWT + OAuth2

**ルール：**
- テスト段階で確定した仕様は変更しない
- データベース設計は一度決めたら変更に慎重
- API仕様は後方互換性を保つ

## データ構造・階層

### マルチテナント階層

```
法人（Corp）
 ├── 事業所（Facility）
 │    ├── 拠点（Location）
 │    ├── サービス種別（設定値）
 │    └── 人員基準（自動適用）
 │
 ├── スタッフ（Staff）
 │    ├── 拠点配置（1拠点 or 複数拠点）
 │    ├── 資格・職種
 │    └── 雇用形態・契約時間
 │
 ├── シフトテンプレート（ShiftTemplate）
 │    ├── 拠点 × 時間帯 × 必要職種
 │    └── 必要人数
 │
 └── シフト申告・勤務体制表（拠点単位で管理）
```

### 集計関係
- 拠点ごとのシフト・勤怠 → 事業所単位で自動集計 → 勤務体制表（実績）生成

## 実装ルール（必読）

### ★ 最重要ルール：コード修正時の安全性

**削除・上書きを防ぐための原則**

1. **機能の「中核部分」は絶対に削除しない**
   - 認証・権限チェック
   - マルチテナント分離ロジック
   - 金銭・法務に関する計算式（常勤換算、人員基準チェック等）
   - 外部API連携（タッチオンタイム）

2. **Phase 1〜6の構造を変えない**
   - 各フェーズは独立した単位として扱う
   - 後ろのフェーズの実装が前のフェーズに依存することはない
   - 逆依存（前 ← 後）は厳禁

3. **ファイル・ディレクトリ構成**
   ```
   /backend
     /src
       /phase1      (事業所登録)
       /phase2      (スタッフ登録)
       /phase3      (シフト提出)
       /phase4      (シフト表作成)
       /phase5      (勤怠連携)
       /phase6      (勤務体制表生成)
       /core        (共通機能・エラーハンドリング・認証)
       /models      (DB スキーマ)
       /utils       (汎用関数)
   
   /frontend
     /src
       /pages
         /phase1
         /phase2
         /phase3
         /phase4
         /phase5
         /phase6
       /components (共通コンポーネント)
       /hooks      (カスタムフック)
       /utils      (フロント汎用関数)
   ```

4. **修正時のチェックリスト**
   ```
   □ 既存テストが全て通るか
   □ マルチテナント分離が壊れていないか
   □ 認証・権限チェックが削除されていないか
   □ 法務計算式（常勤換算等）が正確か
   □ 拠点→事業所の集計ロジックは壊れていないか
   □ タッチオンタイム連携は機能するか
   ```

5. **コード記述時の保護**
   - 重要な計算式には必ず IMMUTABLE コメントを付与
   - マルチテナント分離は explicit filter で実装（whereクエリに tenant_id を含める）
   - 削除系（delete, drop, truncate）は admin権限で管理

### ★ システム動作検証ルール（付け焼き刃禁止）

**重要：UI表示の確認だけでなく、システムとして正しく機能しているか検証する**

1. **データフローの完全性確認**
   - データはどこから取得されるのか（API/localStorage/DB）
   - どのフィールドがどこに保存・参照されるのか
   - Phase間の連携は正しく機能しているか

2. **UI表示 vs 実装ロジックの齟齬確認**
   - 表示される項目（例：勤務曜日）がフロントエンド・バックエンド両方に正しく実装されているか
   - 入力項目（例：スタッフ選択）が手打ちではなく、データ連携で取得されているか
   - 計算結果（例：申告状況カウント）が正しいロジックから導かれているか

3. **検証項目**
   - 【Phase 2】スタッフ登録時に `work_days` などのフィールドが正しく保存されているか
   - 【Phase 3】Phase 2 で登録したスタッフを自動取得して表示できるか（手打ちになっていないか）
   - 【ダッシュボード】各スタッフの情報（勤務曜日・勤務時間など）がPhase 2から正しく取得・表示されているか
   - 【ダッシュボード】申告状況のカウント（例：「申告済み（2日）」）が正しく計算されているか

4. **修正時の確認フロー**
   ```
   表示確認 → データソース確認 → ロジック確認 → 修正 → 全フローで検証
   ```
   表示されているだけでは不十分。データがどこから来ているのか、計算は正しいのかを必ず確認する。

### 修正リクエスト時の指示方法

ユーザーからの修正リクエストには以下の情報を含める：

```
【修正依頼】
- フェーズ: Phase X
- 修正箇所: 具体的なファイルパス・行番号
- 修正内容: 何を変えるか、なぜ変えるか
- 影響範囲: どのテーブル・機能が影響を受けるか
- 確認方法: 修正後にどう検証するか
```

## Phase 4 シフト作成・編集仕様

### シフト作成ボタンの動作

**「シフト作成」ボタンを押したときの処理：**

1. ダッシュボード（Phase 2）に登録されているスタッフ情報を取得
2. スタッフの勤務曜日・勤務時間などをベースに、自動でシフト表を生成
3. 生成されたシフト表が表示される

**重要：** シフト作成ボタンを再度押すと、最新のダッシュボード情報をベースに、シフト表を再生成・初期化する

### シフト編集機能

**シフト表生成後、ページ内での編集：**

1. 生成されたシフト表内で、各セルを自由に手入力で修正可能
2. 編集した値は、ダッシュボード情報とは**独立**して管理される
3. 編集値は状態で保持し、このページ内でのみ有効
4. ページをリロードすると、編集は失われ、最後に生成されたシフト表に戻る

**データフロー：**
- ダッシュボード情報（不変） → シフト作成ボタン → シフト表生成 → 編集（独立） → 表示
